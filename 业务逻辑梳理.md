# 大学生请销假系统 — 业务逻辑梳理

## 1. 系统概述

大学生请销假管理系统，用于高校学生在线提交请假申请、审批流转、销假（返校登记）的全流程数字化管理。系统采用 **Spring Boot 2.7 + Vue 3 (Vant 4) + MySQL 8** 技术栈，移动端优先设计。

---

## 2. 角色体系

| 角色 | 标识 | 权限范围 | 职责 |
|------|------|---------|------|
| **学生** | `STUDENT` | 查看自己的请假记录 | 提交请假申请、撤销申请、发起销假申请、查看审批进度 |
| **辅导员** | `COUNSELOR` | 所管班级 | 第一级审批人（可指定下一级）；直接销假 |
| **副书记** | `SECRETARY` | 本学院 | 第二级审批人（天数 > 3 天时介入）；直接销假 |
| **管理员** | `ADMIN` | 全部数据 | 所有级别审批、直接销假、系统管理 |

## 3. 核心业务流程图

```
┌─────────────────────────────────────────────────────────────────────┐
│                       一、请假流程                                   │
│                                                                      │
│  学生提交请假申请                                                    │
│     │                                                                │
│     ├─ 校验：开始时间 < 结束时间、无时间冲突                          │
│     ├─ 状态 → PENDING                                                │
│     ├─ 创建第1级审批流程（辅导员）                                    │
│     ├─ 若请假天数 > 3 天：额外创建第2级审批流程（副书记）             │
│     └─ 发送通知给第1级审批人                                         │
│                                                                      │
│  辅导员审批（第1级）                                                 │
│     ├─ 驳回 → 请假状态 REJECTED，通知学生                            │
│     └─ 通过                                                          │
│          ├─ 指定了下一级审批人 → 创建第2级审批（副书记），发通知      │
│          ├─ 未指定但已有第2级 → 发送通知给第2级审批人                │
│          └─ 未指定且无第2级 → 请假状态 APPROVED，通知学生            │
│                                                                      │
│  副书记审批（第2级，若有）                                           │
│     ├─ 驳回 → 请假状态 REJECTED，通知学生                            │
│     └─ 通过 → 请假状态 APPROVED，通知学生                            │
│                                                                      │
│  学生撤销（仅限 PENDING 或 APPROVED 状态，且 APPROVED 时必须在       │
│  请假开始时间之前）                                                  │
│     └─ 状态 → CANCELLED，待审批流程也标记 CANCELLED                  │
│                                                                      │
├─────────────────────────────────────────────────────────────────────┤
│                       二、销假流程（两种方式）                         │
│                                                                      │
│  方式A：上级直接销假（辅导员/副书记/管理员操作）                     │
│     ├─ 前提：请假状态为 APPROVED，且尚未销假                          │
│     ├─ 创建 leave_return 记录，自动判断是否逾期                       │
│     ├─ 若有待审批的销假申请 → 自动通过                                │
│     └─ 发送通知给申请人                                              │
│                                                                      │
│  方式B：学生发起销假申请 → 上级审批                                  │
│     ├─ 校验：本人请假、状态 APPROVED、未销假、无待审批销假申请        │
│     ├─ 创建销假申请（PENDING）                                       │
│     ├─ 创建第1级审批流程（辅导员）                                    │
│     ├─ 若原请假天数 > 3 天：额外创建第2级审批流程（副书记）          │
│     └─ 发送通知给第1级审批人                                          │
│                                                                      │
│  销假申请审批（任一级审批人通过即可）                                │
│     ├─ 通过 → 创建 leave_return 记录（判断逾期），其余待审批流程      │
│     │         标记 CANCELLED，通知学生                                │
│     └─ 驳回 → 销假申请 REJECTED，其余待审批标记 CANCELLED，          │
│               学生可重新申请                                          │
│                                                                      │
├─────────────────────────────────────────────────────────────────────┤
│                       三、逾期提醒机制                                │
│                                                                      │
│  ● 系统自动识别即将到期（24小时内）的请假                            │
│  ● 首页弹出销假提醒弹窗                                              │
│  ● 销假时自动判断是否逾期（is_overdue）                               │
└─────────────────────────────────────────────────────────────────────┘
```

## 4. 数据模型与关系

### 4.1 核心实体

```
用户 (user)
  ├─ id              BIGINT (PK)
  ├─ username        VARCHAR(50)   -- 学号/工号（10位数字）
  ├─ password        VARCHAR(255)  -- BCrypt 加密
  ├─ name            VARCHAR(50)   -- 真实姓名
  ├─ role            VARCHAR(20)   -- STUDENT / COUNSELOR / SECRETARY / ADMIN
  ├─ class_id        VARCHAR(50)   -- 班级标识（如 CS2024）
  ├─ college_id      VARCHAR(50)   -- 学院标识（如 CS）
  └─ status          TINYINT       -- 0=禁用, 1=启用

请假申请 (leave_application)
  ├─ id              BIGINT (PK)
  ├─ user_id         BIGINT (FK → user.id)
  ├─ type            VARCHAR(20)   -- SICK / PERSONAL / OTHER
  ├─ start_time      DATETIME
  ├─ end_time        DATETIME
  ├─ days            INT
  ├─ reason          TEXT
  ├─ leave_school    TINYINT       -- 0=不离校, 1=离校
  ├─ status          VARCHAR(20)   -- PENDING / APPROVED / REJECTED / CANCELLED
  └─ attachments     TEXT          -- JSON 数组（文件路径）

审批流程 (approval_process)
  ├─ id              BIGINT (PK)
  ├─ leave_id        BIGINT (FK → leave_application.id)
  ├─ process_type    VARCHAR(20)   -- LEAVE=请假审批, RETURN=销假审批
  ├─ approver_id     BIGINT (FK → user.id)
  ├─ level           INT           -- 1=辅导员, 2=副书记
  ├─ status          VARCHAR(20)   -- PENDING / APPROVED / REJECTED / CANCELLED
  └─ opinion         TEXT

销假记录 (leave_return)
  ├─ id              BIGINT (PK)
  ├─ leave_id        BIGINT (FK → leave_application.id, UNIQUE)  -- 一个请假最多一条销假记录
  ├─ return_time     DATETIME
  ├─ is_overdue      TINYINT       -- 0=按时, 1=逾期
  └─ remark          TEXT

销假申请 (leave_return_application)
  ├─ id              BIGINT (PK)
  ├─ leave_id        BIGINT (FK → leave_application.id)
  ├─ student_id      BIGINT (FK → user.id)
  ├─ status          VARCHAR(20)   -- PENDING / APPROVED / REJECTED
  ├─ remark          VARCHAR(500)
  ├─ approver_id     BIGINT (FK → user.id, nullable)
  └─ approve_opinion VARCHAR(500)

消息通知 (message)
  ├─ id              BIGINT (PK)
  ├─ user_id         BIGINT (FK → user.id)
  ├─ type            VARCHAR(20)   -- APPROVAL / RETURN / SYSTEM
  ├─ is_read         TINYINT       -- 0=未读, 1=已读
  └─ related_id      BIGINT        -- 关联的请假申请ID

操作日志 (operation_log)
  ├─ id              BIGINT (PK)
  ├─ action          VARCHAR(100)
  ├─ module          VARCHAR(50)
  └─ result          VARCHAR(20)   -- SUCCESS / FAIL
```

### 4.2 实体关系图

```
User (1) ──< (N) LeaveApplication          (一个用户可有多条请假)
User (1) ──< (N) ApprovalProcess           (一个用户可审批多条流程)
User (1) ──< (N) LeaveReturnApplication    (一个用户可发起多条销假申请)
User (1) ──< (N) Message                   (一个用户可接收多条消息)
User (1) ──< (N) OperationLog              (一个用户可有多条操作日志)

LeaveApplication (1) ──< (N) ApprovalProcess        (一个请假可有多级审批)
LeaveApplication (1) ──< (N) LeaveReturnApplication (一个请假可有多次销假申请尝试)
LeaveApplication (1) ──? (1) LeaveReturn              (一个请假最多一条销假记录)
```

## 5. 关键业务规则

### 5.1 请假申请规则

| 规则 | 说明 |
|------|------|
| 时间合法性 | `start_time < end_time`，否则拒绝 |
| 时间冲突 | 同一时间段内不能有状态为 PENDING / APPROVED 的请假 |
| 审批级数 | 天数 ≤ 3 天 → 仅辅导员审批；天数 > 3 天 → 辅导员 + 副书记二级审批 |
| 审批人指定 | 辅导员可通过时指定副书记为下一级审批人（覆盖自动规则） |
| 撤销限制 | PENDING 状态随时可撤销；APPROVED 状态只能在请假开始时间之前撤销 |
| 撤销连带 | 撤销后所有待审批的 approval_process 标记为 CANCELLED |

### 5.2 销假规则

| 规则 | 说明 |
|------|------|
| 销假前置条件 | 请假状态必须为 APPROVED 且尚未销假 |
| 逾期判定 | 销假时间晚于请假结束时间 → `is_overdue = 1` |
| 学生销假途径 | 只能通过"发起销假申请"方式，不能直接销假 |
| 上级销假途径 | 辅导员/副书记/管理员可直接销假（无需申请） |
| 销假申请审批模型 | **同级或审批（任意一人通过即可）**，非逐级审批 |
| 销假申请被拒后可重试 | 被驳回后可重新发起销假申请 |

### 5.3 数据权限规则

| 角色 | 可见请假范围 |
|------|-------------|
| STUDENT | 自己的请假记录 |
| COUNSELOR | 同班级（`class_id`）学生的请假记录 |
| SECRETARY | 同学院（`college_id`）学生的请假记录 |
| ADMIN | 全部请假记录 |

## 6. API 接口一览

### 6.1 认证模块 `/auth`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/auth/register` | 注册（学号10位、手机号正则、密码≥6位、BCrypt加密） |
| POST | `/auth/login` | 登录，返回 JWT token + 用户信息 |

### 6.2 用户模块 `/user`

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/user/profile` | 获取当前用户信息 |
| PUT | `/user/profile` | 修改个人信息 |
| PUT | `/user/password` | 修改密码（需旧密码验证） |
| GET | `/user/approvers` | 获取可选审批人列表（仅 SECRETARY 角色） |

### 6.3 请假模块 `/leave`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/leave/apply` | 提交请假申请（自动创建审批流程） |
| GET | `/leave/list` | 查询请假列表（按角色自动过滤数据范围） |
| GET | `/leave/detail/{id}` | 查询请假详情（含审批流程、销假记录、销假申请） |
| PUT | `/leave/cancel/{id}` | 撤销请假申请 |
| POST | `/leave/return/{leaveId}` | 上级直接销假 |
| GET | `/leave/return-reminder` | 获取销假提醒列表 |
| POST | `/leave/return-application/apply/{leaveId}` | 学生发起销假申请 |
| GET | `/leave/return-application/my-list` | 学生查看自己的销假申请记录 |
| GET | `/leave/return-application/pending` | 上级查看待审批的销假申请 |
| GET | `/leave/return-application/approved` | 上级查看已处理的销假申请 |
| POST | `/leave/return-application/approve/{applicationId}` | 审批销假申请 |
| GET | `/leave/return-application/detail/{id}` | 销假申请详情 |
| GET | `/leave/return-records` | 学生查看销假记录 |

### 6.4 审批模块 `/approval`

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/approval/pending` | 获取当前用户的待审批列表（支持 processType 过滤） |
| POST | `/approval/process` | 审批处理（通过/驳回，可指定下一级审批人） |
| GET | `/approval/history` | 获取审批历史 |

## 7. 身份认证与安全

- **认证方式**：JWT（HS512 算法，24 小时过期）
- **密码存储**：BCrypt 加密
- **认证传递**：前端通过 Axios 拦截器在请求头注入 `Authorization: Bearer <token>`
- **后端拦截**：Spring HandlerInterceptor 拦截除 `/auth/login`、`/auth/register`、`/druid/**` 外的所有请求
- **鉴权方式**：服务层按角色字符串做命令式校验（无 Spring Security 注解）
